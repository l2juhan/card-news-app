PLAN.md에서 Phase를 읽고 구현을 진행한다.

## 0. Phase 번호 결정

- `$ARGUMENTS`에 숫자가 있으면 해당 Phase를 실행한다
- `$ARGUMENTS`가 비어있으면 PLAN.md를 읽고 **아직 완료되지 않은 가장 첫 번째 Phase**를 자동으로 찾는다
  - 판단 기준: 체크리스트에 `- [ ]` (미완료 항목)이 남아있는 가장 낮은 번호의 Phase
  - 모든 Phase가 완료되었으면 사용자에게 알리고 종료

## 1. 작업 계획 수립 (사용자 컨펌 전)

실제 구현을 시작하기 **전에** 먼저 계획을 세우고 사용자 승인을 받는다:

1. PLAN.md에서 해당 Phase의 작업 목록을 읽는다
2. 이미 완료된 항목(`- [x]`)은 제외한다
3. 남은 작업들을 **세부 단계(Step)**로 분해한다:

```
Phase {N}: {Phase 목표}

Step 1: {세부 작업 그룹 이름}
  - 작업 A
  - 작업 B
  → 예상 산출물: {생성/수정될 파일 목록}

Step 2: {세부 작업 그룹 이름}
  - 작업 C
  - 작업 D
  → 예상 산출물: {생성/수정될 파일 목록}

...

사용할 SuperClaude 스킬: /sc:implement, /sc:build, ...
병렬 처리 가능 Step: Step 2, 3 (독립적이므로 Agent로 병렬 실행 가능)
```

4. **사용자에게 이 계획을 보여주고 컨펌을 요청**한다
5. 사용자가 수정을 요청하면 계획을 조정한다
6. 사용자가 승인하면 구현을 시작한다

## 2. Git 워크플로우 (컨펌 후, 구현 시작 전)

1. 현재 브랜치 확인 (`git branch --show-current`)
2. 작업 중인 변경사항이 있으면 사용자에게 알리고 확인받기
3. main 브랜치로 이동: `git checkout main`
4. 최신 코드 풀: `git pull origin main`
5. 새 브랜치 생성: `git checkout -b phase-{N}/{간단한-설명}`
   - 예: `phase-1/electron-setup`, `phase-2/template-system`, `phase-3/claude-sdk`

**브랜치 네이밍 규칙:**
- `phase-{번호}/{kebab-case-설명}`
- 설명은 영문 소문자, 하이픈 구분, 3단어 이내

## 3. Step별 구현 진행

계획에서 수립한 Step을 순서대로 실행한다:

### 순차 실행 (기본)
1. 현재 Step의 작업을 순서대로 구현한다
2. 각 작업 시작 시 어떤 작업을 하는지 알려준다
3. **Step 완료 시 중간 컨펌**을 받는다 (아래 "Step 완료 리포트" 참조)
4. 사용자가 승인하면 중간 커밋을 생성하고 다음 Step으로 넘어간다
5. 사용자가 수정을 요청하면 수정사항을 반영하고 다시 리포트 → 컨펌을 반복한다

### Step 완료 리포트
각 Step이 끝나면 아래 형식으로 요약하고 사용자 컨펌을 요청한다:

```
✅ Step {N} 완료: {Step 이름}

작업 내용:
  - {완료한 작업 1}: {생성/수정한 파일}
  - {완료한 작업 2}: {생성/수정한 파일}

변경된 파일:
  - 생성: src/main/index.ts, src/preload/index.ts
  - 수정: package.json

다음 Step: Step {N+1} - {다음 Step 이름}

계속 진행할까요? (수정이 필요하면 말씀해주세요)
```

사용자 응답에 따라:
- **승인** → 커밋 생성 → 다음 Step 진행
- **수정 요청** → 수정사항 반영 → 다시 리포트 → 컨펌 재요청

### 병렬 실행 (독립적인 Step이 있을 때)
- 서로 의존성이 없는 Step은 **Agent를 생성하여 병렬로 실행**할 수 있다
- 예: 여러 독립 컴포넌트 구현, 여러 설정 파일 생성
- Agent 활용 판단 기준:
  - 2개 이상의 독립적인 작업이 있는가?
  - 각 작업이 서로 다른 파일을 수정하는가?
  - 병렬 실행으로 의미 있는 시간 절약이 되는가?
- 병렬 실행 시 `/sc:spawn`을 활용하거나 직접 Agent를 생성한다

### CLAUDE.md 규칙
- CLAUDE.md의 코딩 규칙을 반드시 따른다

## 4. SuperClaude 스킬 활용

작업 성격에 따라 적절한 SuperClaude 스킬을 활용한다:

| 작업 유형 | 사용할 스킬 | 예시 |
|-----------|------------|------|
| 아키텍처/설계 결정 | `/sc:design` | tsconfig 구조, IPC 프로토콜 설계 |
| 코드 구현 | `/sc:implement` | 컴포넌트, 모듈 구현 |
| 빌드/패키징 설정 | `/sc:build` | electron-vite, electron-builder 설정 |
| 테스트/검증 | `/sc:test` | 빌드 테스트, 타입 체크 |
| 복잡한 멀티태스크 | `/sc:spawn` | 여러 독립 컴포넌트 병렬 구현 |
| 문제 해결 | `/sc:troubleshoot` | 빌드 에러, 호환성 이슈 |
| 코드 분석 | `/sc:analyze` | 기존 코드 분석 (instagram-card-news 참조 시) |
| 코드리뷰 | `/review` | Phase 완료 후 최종 코드리뷰 |

**판단 기준:**
- 단순 파일 생성/설정은 직접 수행 (스킬 호출 불필요)
- 설계 판단이 필요한 작업은 `/sc:design` 활용
- 복잡한 구현은 `/sc:implement`로 페르소나 활성화
- 3개 이상 독립 작업이 있으면 `/sc:spawn`으로 병렬 처리 고려
- 참조 프로젝트(`../instagram-card-news/`)의 코드를 가져올 때는 `/sc:analyze`로 먼저 분석
- 외부 라이브러리 사용 시 context7 MCP로 최신 문서를 확인

## 5. 커밋

- **Step 단위로 중간 커밋**을 생성한다 (한 Phase에 여러 커밋)
- 커밋 시 `/commit` 스킬을 사용한다 (커밋 규칙은 해당 스킬에 정의됨)

## 6. 테스트

1. 모든 Step이 끝나면 **완료 기준**에 명시된 검증을 수행한다
2. `/test`로 타입체크 + 린트 + 빌드 + 테스트를 일괄 실행한다
3. 검증 결과를 사용자에게 보고한다
4. 실패 항목이 있으면 수정 후 재검증한다

## 7. 코드리뷰

1. 테스트 통과 후 `/review`로 Phase 전체 변경사항에 대한 코드리뷰를 수행한다
2. Critical 이슈는 자동 수정 → `/commit` → `/test` 재검증
3. Warning 이슈는 사용자 판단에 따라 수정
4. 리뷰 통과 후 다음 단계로 진행한다

## 8. 마무리

1. 리뷰 통과 후 `/end`로 문서 업데이트 + 커밋 + 푸시 + PR 생성을 안내한다
